local_fire_department Step 1: Classification

Gemma 4 26B on Fireworks classifies email into 8 intent categories. Returns category + confidence score (0-1) + reasoning.

memory Step 2: Embedding (AMD ROCm)

e5-large-v2 via AMD GPU generates 1024-dim vector. Enables similarity search against past customer interactions. Graceful fallback to Gemini if AMD unavailable.

edit_note Step 3: Draft + Route

Gemma 4 on Fireworks drafts response using business context, tone rules, and escalation policies. StateGraph routes: auto_reply (confidence > 0.85) or escalate.

info Design Decisions

  • Denormalized user_id on every table — Convex can't efficiently join, so we duplicate for direct user-scoped queries
  • action field replaces status — no separate column for lifecycle state
  • snake_case for all DB fields — consistency with Convex conventions
  • customerEmbeddings uses upsert — one embedding per customer, updated on each interaction

search Query Patterns

  • All queries scoped to user_id via withIndex("by_user")
  • Triage feed uses paginated queries with optional filter/sort/search
  • Similar emails via cosine similarity on embeddings
"Convex schema with denormalized user_id for direct scoping. No joins — Convex doesn't support them efficiently. snake_case everywhere, action field instead of status, upsert pattern for customer embeddings."

balance Trade-offs

Key Design Decisions

Convex vs. Traditional Backend

Chose Convex for reactive subscriptions and managed infra. Trade-off: vendor lock-in, no raw SQL access, limited join support. Mitigation: denormalized schema design.

Fireworks AI vs. OpenAI

Fireworks offers AMD-hosted models and hackathon credits. Trade-off: smaller ecosystem, fewer model options. Mitigation: provider-agnostic abstraction layer (LLM_PROVIDER env var).

Self-hosted AMD ROCm vs. API

Self-hosted for explicit AMD step and zero per-embedding cost. Trade-off: operational overhead, 8hr/day GPU limit, single point of failure. Mitigation: graceful fallback to Gemini.

LangChain StateGraph vs. Custom

StateGraph provides structured workflow orchestration. Trade-off: complex generics require as any casts with Convex. Mitigation: documented workarounds, isolated in agent module.

Direct fetch vs. SDK

Direct HTTP fetch to Fireworks instead of LangChain's ChatFireworks. Trade-off: more boilerplate. Benefit: simpler provider switching, no dependency on LangChain model wrappers.

Denormalized vs. Normalized

Denormalized user_id on every table. Trade-off: data redundancy. Benefit: single-index queries, no joins. Acceptable for Convex's document model at SMB scale.

"Every decision has trade-offs. Convex for reactivity but vendor lock-in. Fireworks for AMD compatibility but smaller ecosystem. Self-hosted AMD for compliance but operational overhead. Denormalized for query performance but data redundancy."
